# 사용자 요구설명서 (통합본 · 수정 반영)

> **1개월** 개발 + EC2 **t2.micro(프리티어)** 제약을 전제로 한 **음식·영양 정보 조회 중심의 AI 서칭 서비스**

## 1. 개요
- **서비스**: 음식·재료의 영양성분/알레르기/보관/가공법을 신속·명확히 제공하는 **AI 서칭 서비스**
- **특징**:  
  - 비로그인 사용자도 AI 질의 가능 → 기록은 익명 저장  
  - 인증/로그인 시점부터 `role=user` 부여, 개인화/히스토리/통계에 반영  
  - 외부 LLM(Gemini) 결과에 **Self-Check** 단계 적용  
  - "단일 질의 → 4섹션 응답(영양/알레르기/보관/가공법) + 출처"
  - 미래 계획: 대화 데이터 전처리 → `음식:영양성분` DB 구축

## 3. 비기능 요구사항(NFR)
- **성능**: p50 ≤ 800ms, p95 ≤ 2000ms(캐시 적중), LLM 포함 p95 ≤ 3500ms
- **오류율**: 전체 ≤ 1%
- **가용성**: 단일 인스턴스(t2.micro), 장애 시 수동 재기동
- **보안/배포**: 운영배포 시 `DEBUG=False`, `ALLOWED_HOSTS` 지정, `SECURE_*`·`CSRF_*` 활성화

---

## 4. 운영 규칙
1) **전역 스로틀**: 익명 20/min, 사용자 60/min  
2) **전역 페이징**: 기본 page_size=20  
3) **캐시 기본**: 검색응답/인기/최근/정적추천 TTL 30–120s  
4) **외부 LLM 호출 가드**: timeout 6s, 토큰 상한, 재시도=0, 서킷브레이커, 짧은 TTL 캐시  
5) **집계는 배치/사전계산만** (실시간 GROUP BY 금지)

### 4.1 Self-Check 규칙
1) **재시도 1회 고정**  
2) **최종 실패 처리**: 빈 값은 숨기지 않고 `"정보 부족"`으로 명시  
3) **로그 일원화**: Self-Check 결과를 InferenceRun에 기록, 관리자 대시보드에서 일/주간 실패율·TOP 위반 규칙 확인 가능

---

## 5. 개발 범위 (1개월)
- **포함**: 4섹션 응답, 인기/최근(사전집계), 자동완성(prefix), 관리자 지표(사전집계), 로그/헬스체크
- **제외**: 벡터DB·개인화 추천·SSR·실시간 대시보드·무거운 워커
- **프런트**: CSR(SPA) 빌드 산출물만 서빙, 요청 중복 취소(AbortController), 타임아웃, 429/408/5xx 에러 UX 표준화

---

## 6. 데이터 모델 특징
- **User.role**: guest/user/admin  
- **비로그인 허용**: Conversation, SearchLog에 user_id/owner_id = NULL 가능  
- **이메일 인증**: `email_verified_at` 컬럼 포함  
- **AI 응답**: Self-Check 구조 반영

### 6.1 InferenceRun (추론 실행 로그)
> 증분 추가 (비파괴, 기존 관계/키 불변)

| 컬럼 | 타입 | NULL | 설명 |
|------|------|------|------|
| check_pass | BOOLEAN | YES | Self-Check 통과 여부 |
| retry_used | BOOLEAN | YES | Self-Check 재시도 여부 |
